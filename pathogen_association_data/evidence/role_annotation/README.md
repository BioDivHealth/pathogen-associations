# WHO Role Annotation

This folder separates biological role review from the core WHO disease-pathogen-host and vector evidence tables.

The files here should not replace the canonical network, vector-screening, competence, GenBank, or WHO DON outputs. Those upstream files provide evidence layers. Role annotation is an interpretation layer that should preserve uncertainty and provenance.

## Files

- `host_role_candidates.csv` is a generated table seeded from the
  master-plus compatibility view for the current active role scope. That view
  is generated from `master_plus_who_host_network.csv` and filtered to the old
  canonical zoonotic association contract.
- `host_role_candidates_summary.csv` is a generated QA summary for the host candidate table.
- `species_host_vector_roster.csv` is a generated collaborator-facing roster with one row per disease plus species or vector taxon. It covers both vectored and non-vectored diseases by combining host rows from the master-plus compatibility view with vector rows from the curated disease-vector table, plus host-vector observation and competence flags where available. Host rows also carry compact modelling-handoff fields from `host_role_assignments.csv` where available (`host_role_assignment`, confidence, review flag, and assignment status); unassigned host rows are marked `host_presence_only`. Vector rows carry compact reviewed/draft assignment fields from `vector_role_assignments.csv` where available (`vector_role_assignment`, confidence, review flag, and assignment status); unassigned vector rows keep those assignment fields blank. `vector_role_hint` remains a separate source/evidence hint rather than the reviewed vector role.
- `species_host_vector_roster_summary.csv` is a generated per-disease QA summary for the species roster.
- `species_host_vector_roster.xlsx` is a generated two-sheet workbook for sharing: `roster` contains the same rows as the CSV, and `column_descriptions` explains each column.
- `host_role_evidence.csv` is the manually curated evidence table for host role claims.
- `host_role_assignments.csv` is the reviewed or draft host-role assignment table.
- `vector_role_evidence.csv` is the manually curated evidence table for vector role claims.
- `vector_role_assignments.csv` is the reviewed or draft vector-role assignment table.
- `vector_role_candidates_who.csv` is a generated WHO-scope vector-role
  candidate surface derived from the conservative WHO-only
  disease-host-vector evidence plus vector-competence annotations. It is not a
  full master-plus/readiness vector-candidate table.
- `vector_role_candidates_who_summary.csv` is the generated per-disease summary
  for `vector_role_candidates_who.csv`.
- `role_modelling_features.csv` is the generated role-annotation feature layer
  consumed by readiness handoff scripts. It derives modelling-facing role proxy,
  bucket, evidence-basis, confidence, review-flag, weight, evidence-tier, and
  missingness fields from the current role assignments plus explicit proxy
  rules under
  `pathogen_association_data/manual/role_annotation/modelling_proxy_rules/`.
- `vector_modelling_features.csv` is the generated vector-side modelling feature
  layer consumed by readiness handoff scripts. It owns compact vector role
  buckets, bucket basis/review flags, vector evidence-tier flags, biological
  evidence tiers, and vector evidence missingness reasons before readiness adds
  SDM availability and model-quality fields.
- `qa/` contains generated role-annotation QA summaries that make the host,
  vector, GenBank-simple, WHO DON, and role-review evidence layers easier to
  inspect before assigning final roles.
- `qa/disease_evidence_readiness.csv` is scoped to the current master-plus WHO
  analysis-unit registry, matching `disease_modelling_readiness_full.csv`
  disease names. Upstream evidence labels outside that registry are listed in
  `qa/disease_evidence_out_of_scope.csv` so WHO DON or GenBank labels do not
  expand the main readiness disease universe.
- `qa/role_modelling_feature_summary.csv` summarizes the generated role
  modelling feature layer by disease, bucket, evidence basis, confidence, and
  review flag.
- `qa/vector_modelling_feature_summary.csv` summarizes the generated vector
  modelling feature layer by disease, vector role bucket, bucket basis,
  evidence tier, evidence level, competence status, and human-biting flag.
- `qa/vector_role_assignments_unmatched_to_roster.csv` is generated by the
  roster build and lists reviewed/draft vector role assignment rows whose
  disease/vector join key is not present in the current vector roster. These are
  usually broad/group-level assignments or taxonomy-grain mismatches that should
  not be silently propagated to species rows.
- Modelling-readiness handoff tables are generated under
  `pathogen_association_data/readiness/`, not in this role-annotation QA folder.
  That folder contains the lean all-disease readiness table, the WHO-focused
  pilot subset, the pilot package CSV/RDS/XLSX bundle, and the full audit
  companion with join diagnostics. Its SDM-species package table uses the
  broader accessible SDM inventory in `sdms/outputs/catalog/` as the SDM
  availability source of truth.

## Role-Review Scope

The current role-review scope is defined as:

```r
(in_gibb_etal == TRUE | in_empres_i == TRUE) &
  Pathogens != "Genus Vesiculovirus"
```

`priority_prototype_status` is retained as descriptive metadata rather than
used as an exclusion filter for role annotation. The GenBank-simple evidence
layer now has a readiness-combined output that binds the original 19-target run
with the expanded readiness run. Evidence-readiness and modelling-readiness
scripts should prefer
`genbank_simple/genbank_readiness_disease_country_summary_standardized.csv`
when it exists, falling back to the older 19-target
`genbank_disease_country_summary_standardized.csv` only for historical reruns.
Rows outside the current role-review scope can remain in the upstream
disease/pathogen tables for provenance and later review. They should not be
deleted just because they are deferred.

The collaborator-facing species roster also excludes broad genus/subgenus source rows with low taxonomic focus:

- `Genus Vesiculovirus`
- `Subgenus Merbecovirus`
- `Subgenus Sarbecovirus`

## Interpretation Boundary

`host_role_candidates.csv` is deliberately conservative. A candidate row means that a host appears in the master-plus compatibility disease-pathogen-host backbone for a role-review-scope disease. It does not mean the host is a reservoir, amplifier, incidental host, or dead-end host.

Final host or vector role labels should be assigned only after source-backed evidence is captured in the corresponding `*_role_evidence.csv` table.

## Manual Curation Conventions

Disease-level role reviews live in `reviews/`. Use `reviews/_role_review_template.md` for new disease reviews, and keep the markdown source log current before adding rows to the manual CSVs.

Group-level taxa can live in the evidence tables when the source only supports a group-level claim, such as `Aves`, `Rodentia`, `Culex spp.`, or `Aedes spp.`. Keep those rows marked for manual review when a downstream species-level interpretation would be tempting. Assignment tables may contain group-level rows only when the assignment is explicitly group-level; do not propagate group evidence to every member species.

For Phase N non-vectored disease reviews, vector findings should normally be recorded in markdown as `not_applicable_non_vectored_scope`. Do not add placeholder vector evidence or vector assignment rows solely to indicate that vectors were not part of the reviewed disease system.

## Regeneration

Use the scoped script paths below for new automation. Historical
`scripts/associations/role_annotation/6_*.R` compatibility wrappers are
archived under `scripts/associations/role_annotation/archive/compatibility_wrappers/`.

Canonical regeneration order is roster, QA, role modelling features, role-gap
source-check candidates, source-check ledger/import, readiness, then
evidence-tier handoff.

Regenerate host role candidates from the repository root with:

```sh
Rscript scripts/associations/role_annotation/roster/01_build_host_role_candidates.R
```

Regenerate the collaborator-facing host/vector species roster with:

```sh
Rscript scripts/associations/role_annotation/roster/02_build_species_host_vector_roster.R
```

Regenerate evidence-readiness QA tables with:

```sh
Rscript scripts/associations/role_annotation/qa/01_build_role_evidence_readiness_qa.R
```

Regenerate role modelling features with:

```sh
Rscript scripts/associations/role_annotation/features/01_build_role_modelling_features.R
```

Host modelling proxy policy is edited in
`pathogen_association_data/manual/role_annotation/modelling_proxy_rules/host_proxy_rules.csv`.
The R rule engine under `scripts/associations/role_annotation/features/rules/`
should stay generic; add disease-specific proxy rows to the CSV unless the
matching language itself needs to change.

Regenerate generated role-gap source-check candidates, then rebuild the
source-check ledger and import accepted rows with:

```sh
Rscript scripts/associations/role_annotation/curation_inputs/04_build_role_gap_source_check_candidates.R
Rscript scripts/associations/role_annotation/source_check/01_build_source_check_decision_ledger.R
Rscript scripts/associations/role_annotation/source_check/02_import_source_checked_role_rows.R
```

Then rebuild the readiness handoff surfaces with:

```sh
Rscript scripts/associations/readiness/01_build_disease_modelling_readiness.R
Rscript scripts/associations/readiness/02_build_modelling_evidence_tiers_handoff.R
```

The Deep Research scripts under `scripts/associations/role_annotation/curation_inputs/`
are on-demand curation tooling. Run them only when preparing, normalizing, or
consolidating new external review batches; routine regeneration starts from the
current roster/features/source-check inputs.
